          <p>The SWG met during the OGC meeting in Palmerston North on 20171206 and discussed several topics.</p>
<h2>
<a aria-hidden="true" class="anchor" href="#terminology" id="user-content-terminology"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Terminology</h2>
<p>The primary source of the terminology in this standard is the ISO Terminology Spreadsheet published 20171019. <strong>Reese Plews has volunteered to check the terminology against this source and to add the correct sources into this document.</strong></p>
<p>The correct syntax for the terminology references is: [SOURCE: ISO 19111:2007, definition 4.8].</p>
<p>Emmanuel Devys identified several terms related to coverages that are defined, but not used in the document. We will delete definitions for items that are not in the document.</p>

<p>There is a significant amount of information in this document that refers to EPSG information that is now obsolete. <strong>Roger Lott will identify this information and delete it and note those deletions in an informative annex to the standard.</strong> This should be done in a way that minimizes changes in the standard.</p>
<h2>
<a aria-hidden="true" class="anchor" href="#axis-order" id="user-content-axis-order"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Axis Order</h2>
<p>The current standard is inconsistent with community practice with regard to Axis order (see <a href="http://www.opengis.net/spec/GeoTIFF/0.0/GTModelTypeGeoKey.geographic" rel="nofollow">http://www.opengis.net/spec/GeoTIFF/0.0/GTModelTypeGeoKey.geographic</a>). This issue is also in the repository (<a href="https://github.com/opengeospatial/geotiff/issues/1">https://github.com/opengeospatial/geotiff/issues/1</a>).</p>
<p><strong>We decided to address this issue (and others in the git repository) in revisions that are expected to happen after the current version of the standard is in the OGC process.</strong></p>

<p>cf. <a href="https://github.com/opengeospatial/geotiff/issues/10">https://github.com/opengeospatial/geotiff/issues/10</a>
A requirement class should be added to express the rules on combination or these tags (and Cardinality of TiePPoints), including tags that can't be used together (e.g. ModelPixelScaleTag + ModelTransformation).
Even and Emmanuel have agreed to elaborate a proposal for this additional requirement class
Note: this is also (at least in part) related to the topic below.</p>
