<a aria-hidden="true" class="anchor" href="#introduction" id="user-content-introduction"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Introduction</h1>
<p>The Spawner ‘spawns’ (generates) almost all of the assets needed for a content model in a web application.
A model consists of inputs, behaviors, relations and some meta properties, such as label and order. Running the Spawner will trigger it to generate a dual layered Javascript model, a dual layered Php module, at least one database table, and create entries in metafiles for namespace, platform configuration and security purposes.</p>

Voilà: <em>nutritious model soup!</em></p>

<a aria-hidden="true" class="anchor" href="#model-config-examples" id="user-content-model-config-examples"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Model config examples</h1>
<p>A minimal Spawner model configuration file:
placed in <strong>/application/modules/default/models/config/MyModestModel.json</strong></p>
<pre><code>{
	"inputs": {
		"name": {}
	}
}
</code></pre>
<p>This will create a MyModestModel model, since the file name defines the model name. It will have an id column, since this is added automatically for every model. It will have a required text type column name, according to your configuration. It will also have the log columns created, modified, author_id, modifier_id.</p>

<h2>
<a aria-hidden="true" class="anchor" href="#batch-mode" id="user-content-batch-mode"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Batch mode</h2>
<p>Calling the Spawner from within a script context, without interactive prompt or semi graphic output:</p>
<pre><code>$ php garp/scripts/garp.php spawn -b --e=testing
</code></pre>
<p>Calling this command triggers the Spawner script. First, all model configs will be read from the configuration files.</p>
<p>It’s also possible to skip certain parts of the Spawn cycle (for instance: to only spawn files, php or the database). For more advanced options, please refer to the Spawn help command:
php garp/scripts/garp.php Spawn help</p>
<h2>
<a aria-hidden="true" class="anchor" href="#the-result" id="user-content-the-result"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>The result</h2>
<p>Per model:</p>
<ul>
<li>A dual layered Javascript model, and a dual layered Php model. See the module configuration property for details.</li>
<li>A database table with columns, indexes and foreign keys</li>
<li>An entry in <strong>content.ini</strong>, for namespace and cms availibility</li>
<li>Entries in <strong>acl.ini</strong> for model-level authorization</li>
</ul>
<p>Also, for the binding models, which act as intermediary in hasAndBelongsToMany relationships:</p>
<ul>
<li>A dual layered Php model: base and extended</li>
<li>A database table with columns, indexes and foreign keys</li>
</ul>
