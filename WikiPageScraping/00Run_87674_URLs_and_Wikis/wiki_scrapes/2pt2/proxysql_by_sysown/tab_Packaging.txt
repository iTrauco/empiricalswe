<a aria-hidden="true" class="anchor" href="#how-do-i-package-proxysql-for-all-the-platforms-it-supports" id="user-content-how-do-i-package-proxysql-for-all-the-platforms-it-supports"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>How do I package ProxySQL for all the platforms it supports?</h3>
<p>The simplest way is to go to the root folder of the repo and run "make packages". This command assumes that you have Docker installed. This will create 2 new packages in the "binaries" folder, one for Ubuntu 14.04 and one for CentOS. Other operating systems are coming soon!</p>
<h4>
<a aria-hidden="true" class="anchor" href="#frequent-errors" id="user-content-frequent-errors"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Frequent errors</h4>
<p>If you're getting an error like this from the packaging script:
"FATA[0000] Error response from daemon: Conflict. The name "ubuntu14_build" is already in use by container d4c8dface7bc. You have to delete (or rename) that container to be able to reuse that name."</p>
<p>.. then all you have to do is run docker rm ubuntu14_build (or the equivalent container name it complains about) and re-run the packaging script.</p>
<h4>
<a aria-hidden="true" class="anchor" href="#how-long-does-the-packaging-take" id="user-content-how-long-does-the-packaging-take"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>How long does the packaging take?</h4>
<p>Each run will take up to 20 minutes. The reason is that it's pulling the fresh package from the source and recompiling ProxySQL against those versions of packages. This can be optimized by reducing to about half of the time if needed, by making use of the Docker layer caching mechanism. Right now we're bypassing it completely in order to be sure that we're always building the latest source tree for ProxySQL into a package.</p>

<p>In theory, it should work on any debian-based operating system. We only tested it on Ubuntu 14.04 so far. However, it should work at least on othe flavours of Linux as well. The most important thing it depends on is Docker's ability to run containers for the supported operating systems on top of another operating system. If this proves to be unfeasible, we'll move away from Docker containers to real VMs on the long run.</p>
