<h4>
<a aria-hidden="true" class="anchor" href="#overview" id="user-content-overview"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Overview</h4>
<p>Purpose of this library is to allow more information to travel with an error with minimal to no changes to the way errors are currently handled. I have tried to identify use-cases that I require and commonalities among different already-existing go error libraries to produce a sensible, simple and approachable API.</p>
<p>I aimed to add the possibility to retain as much information as possible with an error in a structured manner, while still remaining compatible with the built-in error and the way it's usually used.</p>
<h4>
<a aria-hidden="true" class="anchor" href="#justification-for-features" id="user-content-justification-for-features"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Justification for features</h4>
<h5>
<a aria-hidden="true" class="anchor" href="#public-error-message" id="user-content-public-error-message"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Public error message</h5>
<p>Some errors can not be displayed comfortably to an end-users. Some for privacy-issues, some for they just confuse the user. In my opinion this can be alleviated by specifying a "user-friendly" error message at that point where the error is originally created, because that point is where the most amount of information is available. Returning one step up to the calling function, and information could already be lost behind an abstraction.</p>
<h5>
<a aria-hidden="true" class="anchor" href="#errors-having-a-cause" id="user-content-errors-having-a-cause"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Errors having a cause</h5>
<p>Natural. This way when an error needs to be returned to the caller, the reason (an error, if any) can be retained and returned to the caller with the newly-created error.</p>
<h5>
<a aria-hidden="true" class="anchor" href="#errors-having-siblings" id="user-content-errors-having-siblings"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Errors having siblings</h5>
<p>Sometimes it's necessary to return multiple errors. An example is returning all of the validation failures, rather than just one. An array of errors here would complicate the code (internal and function arguments becoming catch-all "interface{}") and automatically make an error return value ([]error) incompatible with the built-in error interface. In this case it would be up to the caller to parse them, whereas in my opinion an error is an just an error, and only if the caller knew how to parse it's multiple errors, should it do so.</p>
<h5>
<a aria-hidden="true" class="anchor" href="#errors-carrying-contextual-data" id="user-content-errors-carrying-contextual-data"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Errors carrying contextual data</h5>
<p>The aim is to unify the way errors carry contextual information, so that it can be processed by a human as well as a machine; contextual information should not be interspersed within the error message, but rather formatted on-demand for humans from data structure or passed further along to some system for storage and later viewing (logs). Also of note is that some information could be vital to the error but might not be available at the point where it should be associated with the error or the other way around. This is where chaining errors with contextual data comes handy.</p>
<h5>
<a aria-hidden="true" class="anchor" href="#errors-having-stack-trace" id="user-content-errors-having-stack-trace"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Errors having stack trace</h5>
<p>Natural. Stack trace easily adds context to an error without any additional work or markup required on part of the error's instantiator. While during debugging it's not strictly necessary, it's bears very little harm (overhead?) to have a quick way to look up from where an error originated.</p>
<h4>
<a aria-hidden="true" class="anchor" href="#reasonings-for-some-of-the-api-choices" id="user-content-reasonings-for-some-of-the-api-choices"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Reasonings for some of the (API) choices</h4>
<p>As can be seem from the commit history, I've swung back and forth between a few different designs, incorporating (in my opinion) the good parts from previous iterations.</p>
<h6>
<a aria-hidden="true" class="anchor" href="#ergo-interfacestruct" id="user-content-ergo-interfacestruct"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Ergo interface/struct:</h6>
<p>It must be possible to embed an ergo error into a user-defined struct, and because error built-in already enables this, disallowing that would be a suicide to this library. Had ergo error been a struct, it wouldn't be possible to type-assert the ergo error out of a user-defined struct, which would have created the need for an "getErgoStruct()" interface polluting the user-defined error and making embedding more difficult.</p>
<p>Only useful conclusion is for ergo error to be an interface, as it can be naturally embedded with minimal invasion and involvement on part of the embedder.</p>
<h6>
<a aria-hidden="true" class="anchor" href="#why-error-interface-has-non-public-fields" id="user-content-why-error-interface-has-non-public-fields"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Why error interface has non-public fields:</h6>
<p>So that some of the underlying ergo struct's fields can be modified. This begs the question "then why not pass around the underlying ergo struct"? Because functions adhering to good-manners(tm) must return the built-in error, not specific types of errors. Thus the caller could no longer call the struct's setter methods. Then "why not make the underlying ergo error struct public?": Because the use cases this library allows would become difficult to impossible if having to handle mixture of plain errors and ergo errors withing one function, simplicity of the API and personal preference. Also ergo error interface is supposed to be used via the library (directly or embedded), and not to be satisfied by externally implementing it, which also ensures the library doesn't blow up with accidentally-satisfied ergo error interfaces. I feel like I'm forgetting some important design decision here.</p>
<h6>
<a aria-hidden="true" class="anchor" href="#constructors" id="user-content-constructors"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z" fill-rule="evenodd"></path></svg></a>Constructors:</h6>
<p>At one fluent interface was used for setting public message, skipping stack frames and adding context. This cannot be, because the ergo interface has to be slim. Current choice was the only way I could think of that's at the same time clean, extensible, and allows setting all settable parameters at construction-time. As a side-"benefit" the OptionAppliers can be used to manually, for example, add context to ergo errors.</p>
